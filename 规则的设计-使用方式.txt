    │
    ▼

    │ no
    ▼
  ③ Pre-LLM 规则评估 (RuleEngine.load_active_rules phase="pre_llm")
     ├─ 用 sess 当前 stage/emotion/回合数 组装 ctx (此时 LLM 还没跑, tags/lang 为空)
     ├─ 按 priority desc 匹配 → 命中后查 fire_policy 是否允许这轮触发
     └─ 命中 transfer_to_human / block_llm → return (本轮不调 LLM)
    │
    ▼
  ④ 跑 LLM:  返回 reply_content / next_stage / customer_emotion / extracted_tags ...
    │
    ▼
  ⑤ 原子更新 session:  SQL 表达式 total_turn+=1, stage_turn (跳了重置为 1 否则 +1), emotion/stage 改写
    │
    ▼
  ⑥ Post-LLM 规则评估 (phase="post_llm")
     ├─ 用 [prev_stage, prev_emotion] + [new_stage, new_emotion, tags, lang] + [新回合数] 组装 ctx
     ├─ 衍生字段: stage_flipped / stage_flipped_to / emotion_degraded (calm→anger 这种)
     ├─ 匹配 → 派发动作
     │     send_text/link/image/video/payment → 写 message + WS push 访客
     │     transfer_to_human → 改 sess.is_human_takeover, 写通知, WS push 给员工

     │     override_reply → 覆盖即将发给访客的 LLM 文本
     └─ 每条命中规则都写一行 session_rule_fires (唯一约束兜底"复读机")
    │
    ▼
  ⑦ LLM 回复消息落库 + WS push 访客 + Mem0 刻录

  规则的"应用范围"如何匹配
  
  RuleEngine.load_active_rules（RuleEngine.py:152 (RuleEngine.py#L152)）一句 SQL：

  WHERE org_id = ?
    AND is_active = TRUE
    AND phase = ?  -- pre_llm or post_llm
    AND (activity_id = ?  OR  activity_id IS NULL)
  ORDER BY priority DESC

  - activity_id = 具体 ID 的规则 → 只对该活动生效
  - activity_id = NULL 的规则 → 该公司所有活动通用
  - 两者并集生效；按 priority 高的先评估；short_circuit=True 命中后停止评估更低优先级
