  瓶颈本质

  --workers 1 不是随便定的,是被嵌入式 Qdrant逼的:

  - glbclient.py:19:AsyncQdrantClient(path=settings.QDRANT_DATA_PATH) —— RAG 知识库,直接读写本地文件目录。
  - main.py:108:mem0 的 vector_store.config.path=MEMORY_DB_PATH —— 访客记忆,又一个本地文件目录。

  嵌入式模式底层是 RocksDB,一个目录同一时刻只能被一个进程打开。uvicorn 多 worker = 多进程,第二个进程一启动就 Storage folder ... already accessed by another instance。所以只能单进程,这台机器的 CPU 多核也吃不满,且无法横向扩展多个后端副本。

  解决方案:Qdrant 改成独立服务(url= 连接)

  核心思路:把"进程内嵌的文件库"换成"独立的 Qdrant 容器",所有后端进程通过网络(http://qdrant:6333)连同一个库。文件锁问题消失 → worker 数随便加 → 甚至能起多个 backend 副本。

  好消息:改动极小。 因为所有 RAG 代码都通过 from glbclient import qdrant_client 用同一个单例,而 Qdrant client 的 API(create_collection / upsert / query_points / delete)在嵌入式和 server 两种模式下完全一样。所以:

  ① 代码改两处
  # glbclient.py —— path 改 url (其余调用方零改动)
  qdrant_client = AsyncQdrantClient(url=settings.QDRANT_URL)   # 如 http://qdrant:6333
  
  # main.py mem0_config 的 vector_store.config —— path 改 host/port
  "config": {
      "collection_name": "saas_visitor_memories",
      "host": settings.QDRANT_HOST,   # "qdrant"
      "port": 6333,
  }   
  配套在 config.py 加 QDRANT_URL / QDRANT_HOST。
  
  ▎ ⚠️  一个隐藏坑:RAG 和 mem0 现在用的是两个不同的本地目录(天然隔离)。换成同一个 Qdrant 实例后,它俩共用一个库,靠 collection 名区分 —— RAG 是 col_{org}_{kb},mem0 是 saas_visitor_memories,不重名,安全。但要记住:从此它们在同一个 Qdrant 里。

  ② compose 加一个服务
    qdrant:
      image: qdrant/qdrant:latest
      restart: unless-stopped
      volumes: 
        - ./data/qdrant:/qdrant/storage   # 持久化,仍然挂卷
      expose:
        - "6333"
  backend depends_on: qdrant,删掉 qdrant_data/、memory_db/ 两个旧卷挂载。

  ③ Dockerfile 解开限制
  CMD [..., "--workers", "4"]   # 或用 gunicorn -k uvicorn.workers.UvicornWorker -w 4
  
  但有几个 worker 化的附带前提(不止换 Qdrant)

  换 Qdrant 是必要条件,但要安全开多 worker,还得注意这几点(否则换了 Qdrant 多 worker 仍有别的坑):

  1. WebSocket 连接管理器是进程内内存(main.py 的 ConnectionManager.active_connections 是个 dict)。多 worker 后,访客的 WS 连在 worker A,但触发下发的请求可能落到 worker B → B 找不到这个连接,消息发不出去。这是多 worker 下比 Qdrant 更隐蔽的坑。
    - 解法:要么后端只跑1 个副本但多 worker仍不够(worker 间也不共享内存),严格说 WS 广播需要跨进程总线(如 Redis pub/sub)才能真正多 worker。
    - 务实折中:Nginx 对 /ws 开 sticky session(ip_hash)让同一访客固定到同一 worker —— 但 uvicorn 多 worker 之间 Nginx 无法区分(同一上游),所以更干净的做法是起多个 backend 容器副本,每个内部单 worker,Nginx 对 /ws 用 ip_hash 粘住。
  2. mem0 / Qdrant 写并发:多 worker 同时写同一 collection,Qdrant server 模式支持,但要确认 mem0 版本对并发的处理。
  3. DEBUG/reload 关掉(已在生产配置处理)。

  我的建议(分阶段)

  - 当前阶段(刚上线、量不大):保持单 worker 完全够用。FastAPI 全异步,单 worker 也能扛相当并发(瓶颈通常在 LLM API 响应,不在 CPU)。你选"先不动"是对的。
  - 量上来后第一步:换独立 Qdrant 容器(上面①②),这是无痛的、随时可做,先把"文件锁"这个硬约束拆掉。
  - 真要多副本扛高并发时:再加 Redis 做 WS 跨进程广播 + 多 backend 副本 + Nginx /ws 粘连。这一步才是大改(要改 ConnectionManager 用 Redis pub/sub)。

  换句话说:EXPOSE 1081 这行没问题,要解的是 --workers 1,而解它的第一颗扣子是独立 Qdrant,真正的天花板是 WS 的进程内状态。
